# Autonomous Loop

The `/loop` skill enables fully autonomous execution of a task queue — from task selection through TDD, validation, handoff, and PR generation — with human-in-the-loop gates at high-risk decision points.

## Overview

Once you have a task queue (generated by `/to-tasks`), the loop executes each task in the correct order, maintaining isolation between tasks through fresh context windows and handoff artifacts.

```
/loop
  ↓
Read task queue (.velocity/artifacts/tasks/)
  ↓
For each task (respecting blocking relationships):
  ├── Open fresh context window
  ├── Run /tdd (red → green → refactor)
  ├── Run /validate (12-point check)
  ├── Run /handoff (artifact for next session)
  ├── [PAUSE if risk score > 70]
  └── Continue to next task
  ↓
Generate PR description
```

## Task Queue Format

The loop reads from `.velocity/artifacts/tasks/[feature].md`:

```markdown
## Task Queue

### TASK-001: Implement RefundRequest domain model

- Status: pending
- Risk: 25
- Blocks: TASK-002, TASK-003
- Estimated: M
- Skills: tdd, domain-model
- Context: orders bounded context

### TASK-002: Add refund API endpoint

- Status: pending
- Risk: 40
- Blocked by: TASK-001
- Estimated: L
- Skills: tdd, api-design
- Context: orders bounded context

### TASK-003: Integrate payment processor refund

- Status: pending
- Risk: 65
- Blocked by: TASK-001
- Estimated: XL
- Skills: tdd, security-design
- Context: payments bounded context
```

## Execution Model

### Context Isolation

Each task runs in a **fresh context window**. The loop passes context to the next task via handoff artifacts — not by maintaining a long conversation. This prevents context contamination between tasks and keeps each session focused.

```
Task 1 session (2000 tokens)
  → Produces handoff-001.md

Task 2 session (fresh, reads handoff-001.md + stack.md + CONTEXT.md)
  → Produces handoff-002.md

Task 3 session (fresh, reads handoff-001.md + handoff-002.md + ...)
```

### Blocking Relationships

The loop respects the blocking graph. TASK-002 and TASK-003 will not start until TASK-001 is complete. The loop identifies the critical path and executes tasks in the optimal order.

### High-Risk Pause

When a task's risk score exceeds the configured threshold (default: 70):

```
⚠ HIGH RISK PAUSE

Task: TASK-003 — Integrate payment processor refund
Risk Score: 65 (threshold: 70)

Reason: External payment API integration with production credentials
Required approval: Engineering Lead

Proceed? [y/N]
```

The loop writes its current state to disk before pausing. Resuming the loop reads the saved state and continues from the last completed task.

### Approval Gates

Configured in `.velocity/guardrails/default.md`, certain categories always require approval regardless of risk score:

```yaml
approval_required:
  - infrastructure_change
  - schema_migration
  - api_contract_change
  - production_data_access
```

### State Persistence

Loop state is persisted at every step:

```
.velocity/artifacts/loop/
├── state.md           # Current status, completed/pending tasks
├── task-001.md        # Task result + handoff
├── task-002.md        # Task result + handoff
└── pr-description.md  # Generated PR description
```

If a session crashes or times out, the next `/loop` invocation reads `state.md` and resumes from the last completed task.

## Loop Configuration

Configure in `.velocity/guardrails/default.md`:

```yaml
loop:
  risk_threshold: 70 # Pause above this score
  max_tasks_per_run: 10 # Safety limit
  fresh_window_per_task: true # Always reset context
  pr_on_completion: true # Generate PR description
  approval_categories: # Always require human sign-off
    - infrastructure_change
    - schema_migration
```

## PR Generation

After all tasks complete, the loop generates a PR description:

```markdown
## Summary

Implements refund support for SettlementBatch items.

## Changes

- TASK-001: RefundRequest domain model with validation
- TASK-002: POST /refunds endpoint with OpenAPI spec
- TASK-003: Payment processor integration (Stripe)

## Testing

- Unit: 24 tests added, all passing
- Integration: 6 new API tests
- Coverage: 87% (threshold: 85%)

## Risk Assessment

- Risk Score: 52 (Medium)
- Approved: TASK-003 by @engineering-lead

## CONTEXT.md Updates

- Added: RefundRequest, RefundWindow, RefundStatus
```

## When Not to Use /loop

The loop is designed for well-specified task queues. Do not use it when:

- Tasks are ambiguous or under-specified
- The feature involves significant architectural decisions (use `/architecture-doc` first)
- You need to explore the problem space (use `/grill-with-docs` first)
- Risk scores are consistently high (address root causes first)
